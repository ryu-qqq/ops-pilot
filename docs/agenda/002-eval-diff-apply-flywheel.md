# 의제 002 — 평가 diff · 반영 · 플라이휠

> 2026-06-05 제기. 다음 세션에서 다룬다. 결론이 아니라 고민을 구조화한 것.

## 어디서 나왔나

작업 탭에서 코드 작업(자동 ingest된 커밋)을 열면 ④ 변경 diff가 비어 있다. 그 diff는 평가 run(work-evaluator)이 worktree에서 바꾼 파일을 보여주는데, 평가자는 읽고 채점만 하지 코드를 고치지 않으니 0건이다. 그래서 "그 커밋의 실제 변경"도, "평가자가 제안하는 변경"도 안 보인다. 여기서 세 갈래 고민이 갈라진다.

## 1. diff를 무엇으로 보여줄까

지금은 평가 run의 (비어 있는) 변경만 보여준다. 바꿀 방향:

- **실제 변경** — 그 작업(커밋 gitRef)이 실제로 무엇을 바꿨는지. 작고 즉효.
- **제안 변경** — 평가자가 "이건 이렇게 됐어야 한다"고 본 변경. 있을 때만.
- 둘을 구분해 보여주고, 제안 변경은 반영할지 말지 고를 수 있어야 한다.

열린 질문: 평가 run이 제안 변경(patch)을 산출하게 할 것인가? 지금 work-evaluator는 채점만 한다. 제안을 내려면 평가자 역할을 넓히거나, 별도 단계가 필요하다.

## 2. "반영"이 정확히 어디에·어떻게

지금 OpsPilot의 apply는 **하네스 자산(`.claude`의 에이전트·스킬·룰)**에 개선안을 반영한다. 그런데 평가 대상이 둘로 갈린다.

- **하네스 자산 자체** — "이 에이전트/스킬을 이렇게 고치자" → 자산에 apply (기존 경로).
- **그 자산이 만든 코드 작업** — "이 코드가 이렇게 됐어야" → 어디에 반영? 이미 커밋·머지된 코드면 반영이 불가하거나 충돌한다.

열린 질문:
- "이게 더 낫다"는 판단이 코드를 향할 때, 반영 대상·시점·방법은? 이미 머지된 경우(새 커밋 제안 / 다음 작업 가이드로 / 반영 포기) 어떻게 처리하나.
- 더 근본적으로 — 평가의 목적이 그 코드를 고치는 게 아니라 **그 코드를 만든 하네스를 고치는 것**일 수 있다. 그렇다면 반영은 늘 하네스로 가고, 코드 diff는 "왜 이 개선이 필요한지"의 근거로만 본다. 이 방향이 맞는지 정해야 한다.

## 3. 플라이휠 — 하네스 엔지니어링에 어떻게 복리가 되나

루프: 작업 → 평가 → (하네스) 개선안 → 반영 → 다음 작업이 나아짐 → 다시 평가.

지금 끊긴 고리: 사람 점수·회고가 "더 나은 프롬프트 추천"으로 환류되지 않고 저장만 된다.

열린 질문:
- 평가·반영이 쌓일수록 하네스가 스스로 좋아지는 구조를 어떻게 닫나.
- 자주 반복되는 지적 → 룰이나 must-reference로 승격?
- 반복 실패 → 자산 버전 후보를 자동 제안?
- 이게 North Star("에이전트가 제대로·일관되게 작동하는지 빨리 판단")의 다음 단계다. 판단을 넘어 **개선이 자동으로 복리**가 되는 것.
