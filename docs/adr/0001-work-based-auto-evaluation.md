# 0001. 작업 기반 자동 평가 — transcript 무상 신호 수집

- 상태: Accepted
- 날짜: 2026-06-01

## 맥락 (Context)

OpsPilot의 North Star는 "만든 자산(에이전트·스킬)이 실작업에서 제대로·일관되게
작동하는지 빨리 판단하도록 돕는가"이다. 기존 평가 경로는 **worktree 재실행 기반
eval**(`feedback/eval-queue.ts`의 `startRun`)인데, 이는 시나리오를 별도로 돌려야 하고
토큰·시간 비용이 들며, 사용자가 그 재실행을 거부하는 경우가 반복됐다.

핵심 질문: **worktree 재실행 없이, 실제 작업 transcript에서 자산 작동 신호를
무상(無償)으로 자동 수집할 수 있는가.**

OpsPilot에는 무상 측정 원칙이 있다 — *trigger·usage 같은 신호는 정답 정의 없이
측정 가능하지만, "품질·가치 판정"은 비용(LLM 호출·사람 스코어)을 요구한다.* 따라서
작업 기반 평가는 무상 측정 가능한 신호에 한정하여 설계한다.

활용 가능한 기존 자산:
- **usage 도메인** (`usage/scan-usage.ts`) — transcript JSONL을 파일 단위로 읽어
  자산 발화(invocation) 집계·cwd 매핑·neverUsed 판정. 본래 전수 재스캔 구조.
- **score 도메인** — `auto-evaluate`의 substring 매칭 선례, 단 `score` 테이블은
  `run_id` FK 강결합.
- **authoring/hooks.ts** — post-commit scan 훅 확장 지점.
- **feedback/eval-queue.ts** — 거부돼 온 worktree 재실행 경로.

이 ADR은 위 자산을 재사용해 **작업 기반 무상 지표를 기본 평가축으로** 세우는 결정이다.
구현은 본 ADR의 다음 단계이며, 여기서는 결정·근거·결과까지만 다룬다.

## 결정 동인 (Decision Drivers)

- **무상성** — 추가 토큰·LLM 호출 없이 측정 가능한가.
- **타당도** — 신호가 "자산이 잘 작동했다"의 정직한 프록시인가, 혼란변수로 오독될
  위험은 없는가.
- **매핑 신뢰도** — transcript ↔ 자산 ↔ 커밋의 연결이 실제로 확인 가능한가.
- **누락 복원력** — 데몬·훅이 죽어도 신호를 빠뜨리지 않는가.
- **기존 루프 리스크** — 현 worktree eval / proposal 생성 루프를 깨지 않고
  점진적으로 전환 가능한가.

## 검토한 옵션 (Considered Options)

### 1. 수집 신호의 범위

- **(A) 발화 + 정정 왕복 (무상 프록시) — 채택**
  발화·neverUsed는 usage 도메인이 이미 집계. 신규 신호 = "자산 발화 후 사용자
  정정(correction) 왕복 횟수". 무상이며 transcript만으로 산출.
- (B) 에러 패턴 (`tool_result is_error`) — 유의미하나 후속으로 미룸.
- (C) LLM 품질 판정 — 무상 원칙 위배(비용), 이번 범위 제외.

트레이드오프: 정정 왕복은 **품질 판정이 아니다.** 정정이 많다고 자산이 나쁜 게
아니다 — 작업 난도·탐색·사용자 변심이 혼란변수다. 따라서 "참고 신호(reference
signal)"로만 라벨한다.

### 2. 평가 단위

- **(A) 세션 (JSONL 파일 1개 = sessionId) — 채택**
  JSONL 파일이 자연 경계이고, scan-usage가 이미 파일 단위로 읽는다.
- (B) 커밋 단위 — transcript↔커밋 매핑이 약한 고리다. JSONL은 `gitBranch`만 갖고
  커밋 SHA 직결이 미확인 → 현 단계에선 비현실적.

### 3. 자동 트리거

- **(A) 주기 스캔 — 채택**
  scan-usage가 본래 전수 재스캔이라, 데몬·훅이 죽어도 다음 스캔에서 누락이
  복원된다. 즉시성은 포기.
- (B) post-commit 훅 확장 — 데몬이 silent하게 죽으면 누락되고, 커밋↔transcript
  매핑 미해결 문제를 그대로 떠안는다 → 비채택.

### 4. 기존 worktree eval의 위치

- **(A) 기본에서 내림 (코드 유지, 명시 호출 시만) — 채택**
  worktree eval은 feedback proposal 생성의 본체라 폐기 시 루프 재설계 위험이 크다.
  코드는 유지하되 **기본 평가축에서 내리고**, 작업 기반 무상 지표를 기본으로 삼는다.
- (B) 완전 폐기 — proposal 루프를 동시에 재설계해야 함 → 리스크 과대, 비채택.

## 결정 (Decision Outcome)

작업 기반 자동 평가를 **다음 4개 결정으로 확정**한다.

1. **수집 신호 = 발화 + 정정 왕복** (무상 프록시). 발화·neverUsed는 기존 usage
   도메인 집계를 그대로 쓰고, 신규로 "자산 발화 후 사용자 정정 왕복 횟수" 파서를
   추가한다. 이 지표는 **"품질 점수"가 아니라 "참고 신호(reference signal)"로
   라벨**한다(타당도 함정 방지). 에러 패턴은 후속 과제로 둔다.

2. **평가 단위 = 세션** (JSONL 1파일 = sessionId = 자연 경계). 커밋 매핑이 약한
   고리이므로 세션을 현실적 경계로 채택한다.

3. **자동 트리거 = 주기 스캔.** 데몬·훅 의존 없이 누락 복원력을 얻는다. 즉시성은
   포기한다. post-commit 훅 확장은 비채택.

4. **worktree eval은 기본 평가축에서 내린다** (코드 유지·명시 호출 시만 동작).
   작업 기반 무상 지표가 기본 평가축이 된다. 폐기가 아닌 점진적 전환이다.

### 설계 세부 결정 (researcher가 열어둔 항목)

- **score 저장 모델 = 신규 테이블.** 기존 `score` 테이블은 `run_id` FK 강결합인데
  작업 기반 지표에는 run이 없다. `score` 테이블에 우겨넣지 않고 **신규 테이블**
  (예: `asset_work_metric` — `assetId`·`sessionId`·발화횟수·정정왕복·기간)을 둔다.
  구현 시 마이그레이션은 `IF NOT EXISTS`로 멱등하게 한다.

- **정정 왕복 경계.** "자산 발화 시점 ~ 다음 자산 발화 또는 세션 끝" 윈도 내의,
  사용자가 타이핑한 메시지(`tool_result`가 아닌 user 메시지) 수로 센다. 한 세션에
  여러 작업이 섞일 때의 경계를 이 윈도로 명시한다.

- **proposal 루프와의 분리.** worktree eval을 기본에서 내리므로, 지표 *수집*과
  개선안 *생성*(proposal)을 분리한다. 작업 기반 지표는 **수집·표시까지만** 담당하고,
  proposal 자동생성은 별도(명시 호출 유지)로 둔다.

- **라벨링 원칙.** UI·데이터 양쪽에서 이 지표가 **"참고 신호(reference signal)"**
  임을 명시한다. "품질 점수"로 오독되지 않게 한다.

## 결과 (Consequences)

### 긍정

- **토큰 0 추가 측정** — transcript만 파싱, LLM 호출·worktree 재실행 없음.
- **사용자가 거부해 온 worktree 재실행에서 탈피** — 기본 경로가 무상 신호로 전환.
- **usage 도메인 재사용** — scan-usage의 파일 단위 읽기·cwd 매핑·집계를 그대로 활용.
- **누락 복원력** — 주기 전수 스캔이라 데몬·훅이 죽어도 다음 스캔에서 복원.

### 부정·위험

- **프록시 타당도 한계** — 정정 왕복은 난도·탐색·변심이 혼란변수다. "참고 신호"로
  한정하지 않으면 "자산 품질"로 오독될 위험 → 라벨링·UI로 강하게 방어해야 함.
- **즉시성 없음** — 주기 스캔이라 작업 직후 즉시 반영되지 않는다.
- **worktree eval 코드 잔존** — 기본에서 내렸지만 코드는 남아 이원적 평가 경로가
  유지된다(유지보수 부담).
- **저장 모델 이원화** — `score`(run 기반)와 신규 `asset_work_metric`(작업 기반)이
  공존한다. 지표 조회·표시 시 두 소스를 다뤄야 한다.

### 후속 과제

- **에러 패턴 신호** — `tool_result is_error` 기반 신호 추가.
- **proposal 루프 재설계** — 수집/생성 분리를 전제로 한 개선안 생성 경로 재정비.
- **커밋 단위 매핑** — 필요 시 transcript↔커밋 SHA 직결을 확보해 커밋 단위 평가로 확장.
